## 🧩 一、主要区别一览表

| 项目           | 主键索引（PRIMARY KEY）   | 普通索引（INDEX / KEY） |
| ------------ | ------------------- | ----------------- |
| **是否唯一**     | ✅ 唯一，且不允许 NULL      | ❌ 可重复，允许 NULL     |
| **数量限制**     | 一个表只能有 **一个主键**     | 可以有多个普通索引         |
| **默认索引类型**   | 聚簇索引（InnoDB）        | 辅助索引              |
| **是否是标识字段**  | 是表的“主标识”，用于唯一定位一条记录 | 仅用于加速查询           |
| **是否自动创建索引** | ✅ 自动创建              | ❌ 需手动添加           |

---

## 📌 二、示例对比

### 主键索引示例：

```sql
CREATE TABLE users (
    id INT PRIMARY KEY,  -- 主键索引，唯一且非空
    name VARCHAR(100)
);
```

### 普通索引示例：

```sql
CREATE TABLE users (
    id INT,
    name VARCHAR(100),
    INDEX idx_name (name)  -- 普通索引，可以重复
);
```

---

## 📘 三、底层实现区别（InnoDB 引擎）

* **主键索引是聚簇索引（Clustered Index）**

  * 表的物理存储顺序按照主键排序。
  * 数据记录就存储在主键索引的叶子节点上。
  * 所以：**主键即数据本身的位置**。

* **普通索引是二级索引（Secondary Index）**

  * 叶子节点存储的是“主键值”，需要再回表去找实际数据（回表操作）。

👉 **性能提示**：

* 查询字段如果能只通过普通索引返回（不回表），叫“**覆盖索引**”，速度很快。

---

## 🧠 四、总结

| 主键索引用于： | 普通索引用于：                |
| ------- | ---------------------- |
| 唯一标识每一行 | 加快查询速度                 |
| 建表时首选   | WHERE、JOIN、ORDER BY 字段 |

---

如果你正在设计一个表结构，一般建议：

1. 每张表都要有一个主键。
2. 根据查询优化需求添加普通索引或组合索引。
