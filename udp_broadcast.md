## ✅ 1. **TCP 和 UDP 的区别**

| 特性     | TCP            | UDP                   |
| ------ | -------------- | --------------------- |
| 连接方式   | 面向连接（需要三次握手）   | 无连接                   |
| 是否可靠   | ✅ 可靠传输（有序、无丢包） | ❌ 不可靠（可能丢包、乱序）        |
| 传输单位   | 字节流（Stream）    | 报文（Datagram）          |
| 是否拥塞控制 | ✅ 有            | ❌ 无                   |
| 适合场景   | 网页、文件传输、邮件等    | 实时音视频、广播、多播、DNS等      |
| 示例协议   | HTTP, FTP, SSH | DNS, DHCP, VoIP, TFTP |

---

## ✅ 2. **UDP 为什么支持广播/一对多？**

### 🔷 UDP 的关键特性是：**无连接 + 面向报文**，这意味着：

* 它不需要知道对方是否存在就可以直接发送。
* 一份数据包可以被多个主机接收（如果网络支持）。

### 🧠 **广播/多播** 本质：

* **广播（Broadcast）**：发送到子网内的所有设备（IP如：`192.168.1.255`）
* **多播（Multicast）**：发送到特定“组”的所有成员（IP如：`224.0.0.1`）

🔸这些只能用 **UDP 实现**，因为 **TCP 是一对一的连接**，不能直接实现多个设备接收同一个流。

---

## ✅ 3. **那 TCP 为什么我可以多个 Tab 页面访问？那不是一对多吗？**

这个问题很聪明，关键点在于：**“一对多”并不等于“广播”或“多播”**

### TCP 是一对一连接，但可以 **开多个连接**：

* 每个 Tab 本质上是单独建立了一个 TCP 连接。
* 它们可能访问相同服务器，但每一个连接是独立的（有自己的三次握手、自己的 socket）。

🔍 举个例子：

你打开 3 个 Tab 访问 `https://www.baidu.com`，发生了什么？

1. 每个 Tab 创建了一个新的 TCP 连接：

   ```
   Client:port1 <--> Server:443
   Client:port2 <--> Server:443
   Client:port3 <--> Server:443
   ```

2. 这是“多个一对一连接”，而不是“一发多人收”的“广播”行为。

## ✅ 4. **UDP 一对多的实际使用场景**

| 应用场景     | 使用方式                    | 是否常用  |
| -------- | ----------------------- | ----- |
| 局域网设备发现  | UDP 广播（255.255.255.255） | ✅ 很常用 |
| IPTV/直播流 | UDP 多播组（224.0.0.x）      | ✅ 常用  |
| 游戏状态同步   | UDP 单播/多播               | ✅ 常用  |
| DNS 查询   | UDP 单播                  | ✅ 常用  |

---

## ✅ 总结：为什么 UDP 能广播，但 TCP 不行？

| 特性        | UDP          | TCP          |
| --------- | ------------ | ------------ |
| 是否连接？     | ❌ 无连接        | ✅ 有连接        |
| 是否允许多设备接收 | ✅ 可以广播/多播    | ❌ 每次只能一个对一个  |
| 应用场景      | 实时通信、组播、设备发现 | 精确通信、传输可靠的数据 |
